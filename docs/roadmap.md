# Universe Monitor — cloud-native roadmap

Progress tracker for the 17-phase GCP plan (`AI Humane Cloud Native Steps`). This is
a **Palantir-style Universe Monitor intelligence platform** deployed as a Google
Cloud-native stack, built on the open-source [World Monitor](https://github.com/koala73/worldmonitor)
application (the same globe/map dashboard, RSS/OSINT feeds, AI briefings, and 39-tool
MCP server).

This repo holds **deployment infrastructure only** — IaC and CI config that stand the
platform up on GCP. It does not fork the application source.

## Phase detail

### Phase 1 — Foundation ✅
- Project: `aihumane-prod` (var `project_id`). APIs enabled in `apis.tf` (Compute, Run,
  Artifact Registry, Cloud Build, SQL, Secret Manager, Storage, DNS, Logging,
  Monitoring, IAP, Vertex AI, BigQuery, Pub/Sub, Scheduler, Redis, VPC Access,
  Service Networking).
- VPC `universe-monitor-vpc` with `public-subnet` / `private-subnet` /
  `management-subnet`, a serverless VPC connector, Cloud NAT, and Private Services
  Access for Cloud SQL — `network.tf`.
- ⬜ Operator: create the project + link billing, then `terraform apply`.

### Phase 2 — Identity ✅
Per-workload service accounts (`uvm-frontend`, `uvm-backend`, `uvm-ai-agent`,
`uvm-cloud-build`, `uvm-terraform`) with least-privilege role grants — `iam.tf`.

### Phase 3 — Source control ✅
Two-repo split: **app** = upstream `koala73/worldmonitor`; **deploy** = this repo. The
plan's `worldmonitor/{frontend,backend,ai-agent,mcp,infrastructure,terraform,docker,docs}`
tree lives upstream for the app and under `gcp/` + `docs/` here for infra.

### Phase 4 — Containerization ✅
Artifact Registry Docker repo `universe-monitor` — `artifact.tf`. Images built by
Cloud Build (`cloudbuild.yaml`): `app` (frontend + api + MCP), plus `ais-relay` and
`redis-rest` support images.

### Phase 5 — Compute ✅
Cloud Run services `uvm-frontend` / `uvm-backend` / `uvm-websocket` / `uvm-auth`
(public, LB-only ingress) and internal `uvm-ais-relay` / `uvm-redis-rest` — `run.tf`.
Long-running AI workers can later move to GKE/GCE (see `docs/architecture.md`).

### Phase 6 — Database ✅
Cloud SQL `universe-monitor-pg` (PostgreSQL 16), **private IP only**, automated
backups + PITR, optional REGIONAL HA (`db_high_availability`). Databases: `users`,
`organizations`, `events`, `knowledge_graph`, `audit_logs`, `tasks`, `documents` — `sql.tf`.

### Phase 7 — Object storage ✅
Buckets `raw-data` / `documents` / `images` / `exports` / `backups`
(`${project}-<name>`), uniform access, lifecycle rules on transient classes — `storage.tf`.

### Phase 8 — Messaging ✅
Pub/Sub topics `osint-feed`, `ai-jobs`, `notifications`, `document-processing`,
`telemetry`, each with a worker pull subscription — `pubsub.tf`.

### Phase 9 — Secrets ✅ containers / ⬜ values
Secret Manager *containers* for every credential in the plan (Anthropic/OpenAI keys,
GitHub token, JWT signing key, OAuth creds, DB password, feed keys) — `secrets.tf`. The
DB password is generated by Terraform; **all other values are added out-of-band**:
```bash
echo -n "sk-ant-..." | gcloud secrets versions add anthropic-api-key --data-file=-
```

### Phase 10 — AI layer ✅
Private **AI worker** (`workers/ai_worker.py`) deployed as a Cloud Run service
(`uvm-ai-worker`, internal ingress, `min-instances = 1`) on the private network —
`workers.tf`. It streaming-pulls the `ai-jobs` / `osint-feed` /
`document-processing` subscriptions and uses Claude (`claude-opus-4-8`, adaptive
thinking, structured output) to turn OSINT into structured records. Optional
self-hosted Ollama service via `enable_ollama = true` — `run.tf`. The broader
autonomous stack (Claude Code agents, MCP servers, Playwright, task queue) layers
on this consumer loop.

### Phase 11 — Search ✅ schema / 🟡 embeddings
**PostgreSQL + pgvector** on the existing Cloud SQL instance —
`gcp/migrations/001_pgvector.sql` creates the extension plus `documents`
(with a `vector(768)` column + IVFFlat index), `entities`, and `relationships`.
Apply it via the seeders job or `psql`. Populating embeddings (a Vertex AI /
OpenAI call into `db.insert_document`) is the remaining step; OpenSearch is a
later scale option.

### Phase 12 — Monitoring ✅
Email notification channel, uptime check on the app host, alert policies for uptime
failure and elevated Cloud Run 5xx — `monitoring.tf`. Error Reporting + Cloud Logging
are on once the APIs are enabled. Set `alert_email` to activate alerting.

### Phase 13 — CI/CD ✅
`gcp/cloudbuild.yaml`: fetch app source → docker build → push to Artifact Registry →
`gcloud run deploy` the app services. ⬜ Operator: create the build trigger.

### Phase 14 — Networking ✅
Reserved static IP, global external HTTPS LB, serverless NEGs per service, Google-managed
TLS cert, HTTP→HTTPS redirect — `loadbalancer.tf`. Cloud DNS zone + A records for
`aihumane.in`, `app.`, `api.`, `docs.`, `admin.` — `dns.tf`. ⬜ Operator: point the
registrar at the zone's name servers (`terraform output dns_name_servers`). Restrict
`admin.` with IAP/VPN.

### Phase 15 — Security ✅
Cloud Armor policy (OWASP XSS/SQLi preconfigured rules + per-IP rate limit), HTTPS LB +
managed TLS, private-IP databases, IAP-only SSH firewall, least-privilege IAM, VPC
firewall rules — `loadbalancer.tf`, `network.tf`, `iam.tf`.

### Phase 16 — Data pipeline ✅
Cloud Scheduler publishes to `osint-feed` / `document-processing`; the AI worker
(`workers/ai_worker.py`) consumes and lands raw payloads in GCS and structured
records in Postgres — `scheduler.tf`, `pubsub.tf`, `workers.tf`. Additional feed
connectors can publish to the same topics.

### Phase 17 — Production architecture ✅
The composed stack (Users → HTTPS LB → Cloud Armor → Cloud Run → private VPC → SQL /
Redis / Storage / Pub/Sub / Secret Manager → private AI workers) is what
`gcp/terraform/` provisions. See `docs/architecture.md`.

## Cost progression (from the plan)
- **MVP (US$100–250/mo):** Cloud Run, Cloud SQL, Cloud Storage, Pub/Sub, Secret Manager.
  Defaults here target this (`deploy_redis` can be off, `min_instances = 0`, ZONAL SQL).
- **Growth (US$500–2,000/mo):** add Redis (`deploy_redis = true`), BigQuery, multiple AI
  workers, monitoring, Cloud Armor.
- **Enterprise:** migrate long-running services to GKE, multi-region, advanced networking.

## How to bring it up

```bash
cd gcp/terraform
cp terraform.tfvars.example terraform.tfvars   # edit project/domain/sizing
terraform init        # needs registry access (provider download)
terraform plan
terraform apply
# then: add secret values, push first image via Cloud Build, point DNS at the LB IP
```

> Provider download (`terraform init`) requires network access to the Terraform
> registry; it is blocked in the authoring sandbox, so plan/apply runs in your own
> environment. `terraform fmt` passes here.
